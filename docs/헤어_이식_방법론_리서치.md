# VARCO 3D → 마스터 VRM 템플릿 헤어 이식 방법론 리서치

> Virtual Avatar 프로젝트의 핵심 기술 난제 정리 + 6주 MVP 로드맵
> 작성일: 2026-04-29 · 대상: 5인 팀 (이현지/조수빈/최윤수/백승우/장동윤)

---

## 1. 문제 정의 — 왜 어려운가

**한 줄로:** VARCO 3D는 사용자 사진을 단일 메시(monolithic GLB)로 출력하는데, 그 안의 "헤어 부분"만 떼어 우리 마스터 VRM 템플릿에 붙이려면 5가지가 한꺼번에 풀려야 한다.

| 도전 과제 | 왜 어려운가 |
|---|---|
| **(1) 메시 분할** | VARCO 출력은 머리·얼굴·몸이 한 덩어리 메시. 어디까지가 헤어인지 라벨이 없음 |
| **(2) 토폴로지 차이** | VRM 표준 헤어는 카드/스트랜드 구조, VARCO 헤어는 임의 형상 → 그대로 붙이면 깨짐 |
| **(3) 리깅·스킨 웨이트** | 마스터 템플릿의 head bone에 정확히 weight를 다시 칠해야 함 |
| **(4) 머터리얼·텍스처** | 헤어 컬러, 하이라이트, 반투명 알파, 셰이더 호환성 모두 재구성 필요 |
| **(5) 물리(SpringBone)** | VTuber 활용을 위해 흔들리는 헤어가 필수. VRM SpringBone 자동 셋업이 까다로움 |

**프로젝트 제약 (솔루션 선택에 결정적):**
- 6주 안에 동작하는 MVP가 나와야 함
- 학생 5인 팀, 헤어 ML 모델을 처음부터 학습할 시간/리소스 없음
- 사용자당 추론 비용 < 3,000원 (BEP 도달용)
- VRM 0.0/1.0 호환 + OBS·VSeeFace 동작 보장
- 헤어는 SpringBone이 동작해야 VTuber로서 가치

→ **결론적으로 학기 내에는 "학습 모델 직접 만드는 길"은 닫혀 있고, "기존 모델 + 잘 조합한 파이프라인"만이 현실적**이다.

## 2. 가능한 접근법 7가지 비교

### A. 이미지 기반 분류 + 프리셋 매칭 (조수빈 현재 방향) ★★★★★

**흐름:** 2D 이미지 → BiSeNet 헤어 마스크 → 시각 임베딩 추출 → 사전 제작된 N종 VRoid 헤어 프리셋과 코사인 유사도 비교 → Top-K 선택 → 마스터 템플릿에 attach

- **핵심 기술:** [BiSeNet face-parsing](https://github.com/zllrunning/face-parsing.PyTorch), [DINOv2](https://huggingface.co/docs/transformers/en/model_doc/dinov2) 임베딩 (CLIP보다 fine-grained 시각 분류에 우수, ViT-B14가 93% 정확도)
- **장점:** 100% VRM 호환 보장, SpringBone이 프리셋에 사전 박혀 있음, 추론 비용 낮음(< 1초·CPU 가능), 6주 안에 충분히 구현 가능
- **단점:** 사용자 헤어와 100% 일치하지 않음, 프리셋 라이브러리 30~50종 사전 제작 필요(이현지님 작업)
- **우리 적합도:** ★★★★★ — **MVP에 채택 권장**
- **예상 구현:** 2~3주

### B. 3D 메시 분할 후 헤어 메시만 transfer ★★★

**흐름:** VARCO GLB → P3-SAM/PartSLIP로 헤어 부위 분할 → 마스터 템플릿 head 위치에 직접 부착 → re-mesh + auto-rig

- **핵심 기술:** [P3-SAM (Native 3D Part Segmentation)](https://arxiv.org/html/2509.06784v3), [SAM 3D Objects](https://github.com/facebookresearch/sam-3d-objects), [Point-SAM (ICLR 2025)](https://github.com/zyc00/Point-SAM)
- **장점:** 사용자 실제 헤어 형상 유지 가능
- **단점:** 분할된 메시의 토폴로지가 무작위 → 깨짐·구멍 빈번, SpringBone 자동 적용 불가, GPU 추론 비싸짐, 라이센싱(SAM3D는 Meta 라이선스 확인 필요)
- **우리 적합도:** ★★★ — **Phase 2 후보**
- **예상 구현:** 4~6주 단독

### C. 헤어 전문 생성 모델 (Im2Haircut, Gaussian Haircut) ★★★

**흐름:** 단일 이미지 → strand 기반 3D 헤어 재구성 → 우리 VRM에 변환

- **핵심 기술:** [Im2Haircut (ICCV 2025)](https://im2haircut.is.tue.mpg.de/) — 단일 뷰에서 strand 기반 재구성 가능, [Gaussian Haircut (ECCV 2024)](https://eth-ait.github.io/GaussianHaircut/) — 멀티뷰 필수, [Towards Unified 3D Hair Reconstruction (SIGGRAPH Asia 2024)](https://dl.acm.org/doi/10.1145/3680528.3687597)
- **장점:** 최신 SOTA 품질, 사용자 헤어 거의 그대로 재구성
- **단점:** strand 출력 → VRM 카드 헤어로 변환하는 추가 파이프라인 필요(없음), 모델 크고 GPU 비쌈, 라이선스 비상업/연구용 다수
- **우리 적합도:** ★★★ — **Phase 3 비전**
- **예상 구현:** 8주+ 단독, 학생 팀에는 무리

### D. Wrap deformation / Non-rigid registration ★★★★

**흐름:** 마스터 템플릿의 표준 헤어 메시를 VARCO 헤어 형상에 맞춰 변형(wrap) → 토폴로지는 마스터 그대로 유지하면서 외형만 사용자화

- **핵심 기술:** [Bolkart's TemplateFitting](https://github.com/TimoBolkart/TemplateFitting), [Houdini Topo Transfer](https://www.sidefx.com/docs/houdini/nodes/sop/topotransfer.html), [Deformation Transfer](https://github.com/chand81/Deformation-Transfer)
- **장점:** 토폴로지 유지 → SpringBone·블렌드쉐입·웨이트가 깨지지 않음, A보다 사용자 일치도 높음
- **단점:** 비강체 정합 알고리즘이 까다롭고 실패 케이스(헤어가 길거나 비대칭일 때) 많음, Blender bpy로 자동화하면 복잡
- **우리 적합도:** ★★★★ — **Phase 2 핵심 후보 (A의 정확도 보강)**
- **예상 구현:** 3~4주

### E. Latent 헤어 임베딩 → VRoid 파라미터 매핑 ★★

**흐름:** 헤어 이미지를 latent 공간에 인코딩 → VRoid 파라미터 공간으로 회귀

- **장점:** 한 번 학습되면 빠름
- **단점:** **VRoid 헤어 파라미터 ↔ 이미지 학습 데이터셋이 없음** → 처음부터 만들어야 함
- **우리 적합도:** ★★ — 데이터 비용이 학기 프로젝트에 부적합

### F. 헤어 가이드 스트로크 추출 + 절차적 헤어 합성 ★★★

**흐름:** 2D에서 헤어 흐름선(orientation map) 추출 → Blender Hair Curves로 절차적 합성 → VRM 카드 변환

- **장점:** 사용자 헤어 흐름 반영, Blender 표준 도구 사용
- **단점:** 절차적 헤어를 VRM 카드 메시로 굽는 단계가 수작업, 자동화 난이도 높음
- **우리 적합도:** ★★★ — Phase 2 보조

### G. 사용자 선택 + AI 추천 하이브리드 UI ★★★★★

**흐름:** A의 결과로 Top-3 후보 헤어 프리셋을 사용자에게 보여주고 직접 고르게 함 (+ "다른 헤어 보기" 옵션)

- **장점:** 100% 정답이 아니어도 사용자 만족도 높음, 데이터 수집(어떤 추천이 선택됐는지) 가능
- **단점:** UX 흐름이 한 단계 늘어남
- **우리 적합도:** ★★★★★ — **A와 함께 MVP 필수**
- **예상 구현:** 1주 (백승우 웹뷰어 위에)

## 3. 단계별 로드맵 — 우리 팀 상황 기준

### MVP (지금~학기말, 6주) — **A + G 하이브리드 채택**

**한 줄:** "BiSeNet으로 헤어 마스크 → DINOv2로 시각 임베딩 → VRoid 프리셋 30종과 매칭 → 사용자에게 Top-3 제시 + 직접 선택"

**왜 이 조합인가:**
- 6주 안에 5인이 분담해서 만들 수 있는 유일한 길
- 100% VRM 호환 + SpringBone 사전 박힌 안전한 아웃풋
- 사용자 만족도는 G(사용자 선택)가 보강 → "추천이 100% 정확하지 않아도 OK"라는 UX 약속
- 추후 D(Wrap deformation)로 정확도 업그레이드할 수 있는 구조

**평가 기준:**
- Top-3 안에 사용자가 만족하는 헤어가 들어갈 확률 ≥ 70% (베타 50명 정성 평가)
- 헤어 색상 ΔE76 (LAB 색공간) ≤ 15
- 1장 처리 시간 ≤ 5초 (CPU·GPU 혼용)

### Phase 2 (학기 후 ~ 6개월) — **D + B 보강**

**목표:** "추천이 마음에 안 드는 사용자"를 위한 정밀 모드 추가

- (D) Wrap deformation으로 마스터 헤어를 사용자 헤어 형상에 변형 — 토폴로지 유지하면서 외형 사용자화
- (B) P3-SAM으로 VARCO 헤어 메시 분할 → 색상·하이라이트 더 정밀하게 추출
- 헤어 프리셋 라이브러리 50 → 100종으로 확장
- 자체 데이터 수집: MVP에서 사용자가 어떤 추천을 선택했는지 로그 → 다음 학습 데이터

### Phase 3 (1년+) — **C 또는 자체 학습**

- Im2Haircut 라이선스 확보 또는 비슷한 strand 기반 모델 자체 학습
- strand → VRM 카드 헤어 변환 파이프라인 자체 개발
- 텍스트 프롬프트 헤어 편집 ("앞머리 더 길게", "웨이브 추가") — HeadEvolver 계열 활용

## 4. MVP 워크플로우 상세 — 의사코드까지

```
[입력] 사용자 정면 사진 (PNG/JPG, 512x512+)
   ↓
[1] BiSeNet face-parsing
    pip: face-parsing.PyTorch (zllrunning)
    출력: hair_mask (binary), face_mask, eyes_mask, ...
    예상 시간: ~150ms (GPU), ~600ms (CPU)
   ↓
[2] hair_mask 후처리
    - Hair matting (FBA_Matting) 또는 GuidedFilter로 경계 부드럽게
    - 모자/안경이 hair로 잘못 분류된 경우 → BiSeNet의 hat/eye_glass 채널과 XOR
    - 결과: alpha-mask + cropped_hair_image
   ↓
[3] 헤어 색상 추출
    - LAB 색공간으로 변환
    - hair_mask 영역에서 K-means(K=3) → primary_color, highlight_color, shadow_color
    - 평균 한 색이 아니라 그라데이션 정보 보존
   ↓
[4] 헤어 스타일 임베딩
    - cropped_hair_image를 224x224로 리사이즈
    - DINOv2-ViT-B14로 768-dim 임베딩 추출
    - GPU 추론 ~50ms
   ↓
[5] 프리셋 매칭
    - 사전 등록된 30~50종 VRoid 헤어 프리셋 임베딩(Postgres pgvector 저장)
    - 코사인 유사도 Top-3 선택
    - 선택지에 길이/볼륨/앞머리 메타데이터 함께 표시
   ↓
[6] 사용자 선택 (UX)
    - Top-3을 마스터 템플릿에 한 종씩 attach해서 미리보기 3장 렌더
    - 사용자가 클릭으로 1개 선택 ("내 머리와 가장 비슷")
    - "다른 스타일 보기" → 전체 50종 갤러리
   ↓
[7] 마스터 템플릿에 헤어 attach
    - 선택된 헤어 프리셋의 mesh를 head bone에 parent
    - SpringBone 설정은 프리셋에 사전 박혀 있음 → 그대로 사용
    - 머티리얼 albedo에 [3]의 primary_color 적용
    - 슬라이더로 길이/볼륨 미세조정 (블렌드쉐입 0~1)
   ↓
[8] Unity headless에서 UniVRM 익스포트 → VRM 0.0 출력
```

**각 단계 라이브러리 / 예상 정확도 / 실패 케이스 정리**

| 단계 | 라이브러리 | 예상 정확도 | 실패 케이스 / 회피 |
|---|---|---|---|
| 1 | face-parsing.PyTorch (BiSeNet) | mIoU 80%+ | 어두운 배경 → 임계값 조정 / 옆모습 → 정면 강제 |
| 2 | FBA_Matting / GuidedFilter | 시각적 깔끔 | 바람에 흩날린 헤어 → 사용자에게 사진 가이드 |
| 3 | OpenCV K-means LAB | ΔE76 < 15 | 염색·하이라이트 강한 경우 → 3색 보존 + 사용자 색 조정 슬라이더 |
| 4 | DINOv2 ViT-B14 (HuggingFace) | 분류 정확도 ~93% | 일반 사진 학습 → 헤어 특화 fine-tune은 Phase 2 |
| 5 | Postgres pgvector | < 50ms | 임계값 낮으면 "Top-1 자동 적용", 높으면 "사용자 선택 강제" |
| 6 | Three.js 웹뷰어 (백승우) | 사용자 만족 70%+ | 50종 다 봐도 마음에 안 드는 사용자 → "기본 헤어로 시작" 옵션 |
| 7 | UniVRM (Unity headless) | 100% VRM 호환 | 머티리얼 호환성 사전 검증된 프리셋만 사용 |

## 5. 헤어 프리셋 라이브러리 구축 가이드

**MVP 권장 30종, Phase 2까지 50~100종.**

**분류 축 (직교):**
- 길이: short / medium / long / extra-long (4)
- 스타일: straight / wavy / curly (3)
- 앞머리: yes / no (2)
- 볼륨: thin / normal / volume (3)

→ 전수 조합은 72종이지만 자주 쓰이지 않는 조합 제외하면 30종으로 80% 커버 가능.

**각 프리셋이 가져야 할 메타데이터:**

```json
{
  "id": "hair_001",
  "name": "Medium Wavy Bangs",
  "thumbnail": "s3://.../hair_001_thumb.png",
  "preview_render": "s3://.../hair_001_preview.png",
  "vrm_mesh_path": "templates/hair/hair_001.glb",
  "tags": { "length": "medium", "style": "wavy", "bangs": true, "volume": "normal" },
  "blendshapes": ["length_blend", "volume_blend", "bangs_blend"],
  "spring_bone_setup": {
    "root_bones": ["hair_root_l", "hair_root_r"],
    "stiffness": 0.8, "drag_force": 0.4, "gravity_power": 0.3
  },
  "default_color": "#3a2820",
  "embedding": [0.123, -0.045, ...]  // DINOv2 768-dim
}
```

**라이선스 주의:**
- VRoid 무료 에셋 중 "VRM 0.0 사용 가능" + "상업 이용 가능" 라이선스만 채택
- 상업용 헤어 에셋 마켓플레이스(BOOTH 등)에서 5~10개 라이선스 구매 권장 (단가 1~3만원)
- 이현지 팀원이 Blender로 일부 자체 제작 (이미 베이스 모델 만들어본 경험 있음)

## 6. 리스크 & 회피 전략

| 리스크 | 회피 |
|---|---|
| "내 머리와 다르다" 인식 | Top-3 + 전체 갤러리로 사용자 선택권 보장. "베타 단계"라는 UX 명시 |
| 헤어 색이 단순 평균 → 부정확 | K-means 3색(주색·하이라이트·그림자) 추출, 머티리얼에 그라데이션 텍스처 적용 |
| 모자/안경이 헤어로 오인 | BiSeNet의 hat/eye_glass 채널을 명시적으로 hair에서 제외 |
| 프리셋에 SpringBone 누락 | 라이브러리 등록 시 SpringBone 동작 자동 테스트 (Unity headless에서 5초 시뮬레이션 후 캡처) |
| VRoid 라이선스 위반 | 등록 시 라이선스 메타데이터 필수 입력, 자체 제작 우선 |
| BiSeNet inference 비용 | CPU 추론으로도 ~600ms 가능 → MVP는 단일 EC2 t3.xlarge로 충분 |
| 사용자가 50종 다 봐도 만족 못 함 | "다음 업데이트에서 더 추가됩니다" 메시지 + 피드백 폼으로 헤어 수요 수집 |

```python
# pseudo-code: 매칭 API
import torch
from transformers import AutoModel, AutoImageProcessor

dino = AutoModel.from_pretrained("facebook/dinov2-base")
proc = AutoImageProcessor.from_pretrained("facebook/dinov2-base")

def hair_embedding(hair_crop_pil):
    inputs = proc(images=hair_crop_pil, return_tensors="pt")
    with torch.no_grad():
        out = dino(**inputs).last_hidden_state.mean(dim=1)  # (1, 768)
    return out[0].cpu().numpy()

# pgvector 매칭
def topk_presets(emb, k=3):
    return db.fetchall("""
        SELECT id, name, thumbnail, 1 - (embedding <=> %s) AS sim
        FROM hair_presets
        ORDER BY embedding <=> %s
        LIMIT %s
    """, (emb, emb, k))
```

## 8. 결론 요약

> **MVP에 채택할 단 하나의 길:** 조수빈 팀원이 이미 가닥 잡은 "프리셋 매칭" 방향이 우리 6주 + 5인 + VRM 호환 제약 안에서 유일하게 현실적인 답이다. 단, **Top-1 자동 적용이 아닌 Top-3 + 사용자 선택 UX**로 보강해 정확도 부족을 사용자 통제권으로 메운다.

> **정확도 업그레이드 경로는 이미 보인다:** Phase 2에서 D(Wrap deformation)로 마스터 헤어를 사용자 헤어 형상에 변형하고, Phase 3에서 Im2Haircut 같은 strand 기반 모델로 진화하면 된다. MVP 구조를 잘 짜두면 매칭 모듈만 교체해서 단계적으로 올라갈 수 있다.

> **헤어 프리셋 라이브러리 30종이 진짜 자산이 된다.** 이현지 팀원의 마스터 템플릿 + 30종 헤어 프리셋 + 각 프리셋의 SpringBone 사전 설정이 우리 제품의 진짜 진입 장벽이고, 경쟁사가 따라오기 어려운 부분이다.
